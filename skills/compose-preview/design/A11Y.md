# Accessibility (a11y)

Two complementary checks. Always do both before shipping a UI change:

## 1. Visual review (every change)

Read the rendered PNG. The renderer captures the actual composition, so
contrast, hit-target size, truncation, RTL mirroring, font-scale overflow,
night-mode colors, and Wear edge clipping are all inspectable directly.
`compose-preview show --json` surfaces `changed: true` per capture so you
only re-read what moved.

For broad coverage, run the preview through font-scale and night-mode
multi-preview meta-annotations (e.g. `@PreviewFontScale`,
`@PreviewLightDark`, `@WearPreviewFontScales`) — each variant lands as its
own entry with a unique id.

## 2. ATF (Accessibility Test Framework) checks

ATF is **opt-in** — by default Android renders skip the post-capture ATF + hierarchy walk so a
plain `./gradlew renderAllPreviews` stays cheap. Opt in either through the typed plugin DSL or the
matching Gradle property:

```kotlin
// build.gradle.kts — always-on for this module
composePreview {
    previewExtensions {
        a11y { enableAllChecks() }
    }
}
```

```sh
# one-shot — same property the CLI's `a11y` command sets internally
./gradlew renderAllPreviews -PcomposePreview.previewExtensions.a11y.enableAllChecks=true
```

The CLI's `compose-preview a11y` command is a thin wrapper that forwards this property and reads
the canned report — equivalent to `compose-preview show --with-extension a11y` plus the report
rendering. Internally it's a `ReportCommand` bound to the `a11y` id from the strategy registry;
`compose-preview extensions list` enumerates every registered extension and their enable
properties. Findings (touch-target size, contrast, missing `contentDescription`, etc.) land in
`accessibility.json` and an annotated PNG with numbered badges. Findings never fail the build —
they're surfaced via the CLI / VS Code, not as a gate. Use `--fail-on errors` on the CLI when you
want a non-zero exit code locally or in CI.

```sh
compose-preview a11y                       # human-readable findings (opts in automatically)
compose-preview a11y --json --changed-only # for re-render loops
compose-preview a11y --fail-on errors      # non-zero on ERROR-level

# Generic equivalent — useful when stacking with other extensions:
compose-preview show --with-extension a11y --json
```

JSON entries include `a11yFindings[]` (level/type/message/viewDescription) and
`a11yAnnotatedPath`. Read the annotated PNG to map numbered badges back to
findings. ATF needs live semantics, so when it's enabled the renderer keeps `LocalInspectionMode =
false` (which is also the default for the other render paths); the cost is just the post-capture
walk and JSON write, skipped entirely when ATF is off.

### Saving the flag combination as a profile

`compose-preview profile <path.json>` runs a captured flag combination. Use it when a team has a
standard "our auth module's a11y check" recipe and you don't want to re-type the same flags every
run — a profile is `extensions` + `filter` + `failOn` + a chosen `report` extension, executed
through the same `ReportCommand` `compose-preview a11y` uses, so the output and exit codes match.

```json
{
  "schema": "compose-preview-profile/v1",
  "extensions": ["a11y"],
  "filter": {
    "module": ":samples:android",
    "idSubstring": "Home",
    "changedOnly": true
  },
  "failOn": {"a11y": "errors"},
  "report": "a11y"
}
```

```sh
compose-preview profile auth-a11y.json              # exits non-zero on ERROR-level findings
compose-preview profile auth-a11y.json --json       # last-write-wins; overrides profile output
```

Profiles are deliberately a "captured flag set," not a programming model. For predicate filters,
per-result hooks, or multi-extension orchestration, see the Kotlin scripting follow-up tracked in
[issue #1084](https://github.com/yschimke/compose-ai-tools/issues/1084).
